FCFS (First Come First Serve)

Description

The simplest disk scheduling algorithm that services requests in the order they arrive in the queue.

Like a first-come-first-served queue at a ticket counter, it processes requests in their arrival order without any prioritization.

How It Works

  1. The disk head starts at its initial position
  2. Moves to the first request in the queue
  3. Then moves to the next request in the order they arrived
  4. Continues until all requests are serviced

Advantages

  • Simple to understand and implement
  • No starvation - every request gets serviced
  • Fair to all requests

Disadvantages

  • Poor performance in terms of total seek time
  • Doesn't take advantage of locality of reference
  • Can result in wild swings of the disk arm

Example

Requests: 98, 183, 37, 122, 14, 124, 65, 67

Head starts at: 53

Order: 53 → 98 → 183 → 37 → 122 → 14 → 124 → 65 → 67

Total seek distance: 640 cylinders

Performance Characteristics:

Time Complexity: O(n)

Space Complexity: O(1)

Average Seek Time: High (depends on request order)

Description

Selects the request with the minimum seek time from the current head position, minimizing overall seek time.

Similar to how an efficient waiter might serve tables closest to their current position first.

How It Works

  1. The disk head starts at its initial position
  2. Calculates seek distance to all pending requests
  3. Selects the request with the shortest seek time
  4. Moves to that request and services it
  5. Repeats the process until all requests are serviced

Advantages

  • Reduces total seek time compared to FCFS
  • Better average response time
  • More efficient than FCFS

Disadvantages

  • Starvation possible for requests far from head position
  • Not guaranteed to have minimum seek time
  • Requires dynamic calculation at each step

Example

Requests: 98, 183, 37, 122, 14, 124, 65, 67

Head starts at: 53

Order: 53 → 65 → 67 → 37 → 14 → 98 → 122 → 124 → 183

Total seek distance: 236 cylinders

Performance Characteristics:

Time Complexity: O(n²)

Space Complexity: O(n)

Average Seek Time: Better than FCFS

Description

The disk arm starts at one end of the disk and moves toward the other end, servicing requests as it goes.

Like an elevator moving from the ground floor to the top floor, stopping at requested floors along the way.

How It Works

  1. The disk head starts at initial position with a direction (left or right)
  2. Moves in that direction servicing all requests encountered
  3. When it reaches the end of the disk (0 or max track), it reverses direction
  4. Services requests while moving in the opposite direction
  5. Continues this back-and-forth movement

Advantages

  • No starvation - all requests eventually serviced
  • Better performance than FCFS/SSTF for heavy loads
  • Predictable service pattern

Disadvantages

  • Long wait time for requests just visited by the head
  • Not most efficient for light loads
  • May do unnecessary seeks to disk ends

Example

Requests: 98, 183, 37, 122, 14, 124, 65, 67

Head starts at: 53 (moving left)

Order: 53 → 37 → 14 → 0 → 65 → 67 → 98 → 122 → 124 → 183

Total seek distance: 236 cylinders

Performance Characteristics:

Time Complexity: O(n log n) for sorting

Space Complexity: O(n)

Average Seek Time: Good for heavy loads

Description

Similar to SCAN but the arm only goes as far as the last request in each direction, making it more efficient.

Like an elevator that only goes as high as the highest requested floor and as low as the lowest requested floor.

How It Works

  1. The disk head starts at initial position with a direction (left or right)
  2. Moves in that direction servicing all requests encountered
  3. When it reaches the last request in that direction, it reverses direction
  4. Services requests while moving in the opposite direction
  5. Doesn't go to the end of disk unless there are requests there

Advantages

  • Reduces unnecessary movement compared to SCAN
  • No starvation - all requests eventually serviced
  • More efficient than SCAN in most cases

Disadvantages

  • Slightly more complex to implement than SCAN
  • Still has variable wait times depending on direction

Space Complexity: O(n)

Average Seek Time: Better than SCAN

Description

Services requests in one direction and quickly returns to the beginning to service more requests, providing more uniform wait time.

Like a circular shuttle bus that goes in one direction and then quickly returns to the start point.

How It Works

  1. The disk head starts at initial position with a direction (typically right)
  2. Moves in that direction servicing all requests encountered
  3. When it reaches the end of the disk, it jumps to the opposite end
  4. Continues servicing requests in the same direction
  5. Doesn't service requests during the return jump

Advantages

  • More uniform wait time than SCAN
  • Good for systems with many requests
  • Predictable service pattern

Disadvantages

  • Wastes time by moving to the end when not needed
  • Longer average response time than LOOK
  • Doesn't service requests during return jump

Example

Requests: 98, 183, 37, 122, 14, 124, 65, 67

Head starts at: 53 (moving right)

Order: 53 → 65 → 67 → 98 → 122 → 124 → 183 → 200 → 0 → 14 → 37

Total seek distance: 382 cylinders

Performance Characteristics:

Time Complexity: O(n log n) for sorting

Space Complexity: O(n)

Average Seek Time: Good for uniform access

Description

Variant of C-SCAN that only goes as far as the last request in each direction, making it more efficient.

Like a modified elevator that only goes as high as needed and then jumps back to the lowest requested floor.

How It Works

  1. The disk head starts at initial position with a direction (typically right)
  2. Moves in that direction servicing all requests encountered
  3. When it reaches the last request in that direction, it jumps to the first request in the opposite direction
  4. Continues servicing requests in the same direction
  5. Doesn't go to the end of disk unless there are requests there

Advantages

  • More efficient than C-SCAN
  • Uniform wait time like C-SCAN
  • Reduces unnecessary disk arm movement

Disadvantages

  • Slightly more complex to implement than C-SCAN
  • Still has jump overhead like C-SCAN

Requests: 98, 183, 37, 122, 14, 124, 65, 67

Time Complexity: O(n log n) for sorting

Space Complexity: O(n)

Average Seek Time: Best among SCAN variants
